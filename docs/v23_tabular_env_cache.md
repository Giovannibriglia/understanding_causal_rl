# v23 — TabularSepsisEnv tensor cache

## What and why

`TabularSepsisEnv.__init__` was rebuilding two large constant tensors
on every instance:

* `transition`: shape `(LATENT_STATES, N_ACTIONS, LATENT_STATES) =
  (1440, 8, 1440)`, ~16.6 M floats.  Built by a Python loop over
  `LATENT_STATES × N_ACTIONS = 11,520` entries.
* `reward_probs`: shape `(LATENT_STATES, N_ACTIONS, 2) = (1440, 8, 2)`,
  ~23 K floats.  Same loop structure.

Total build time: ~200 ms on CPU per env.  `BenchmarkRunner._run_offline`
constructs a fresh env at every eval / oracle / gap-metric checkpoint
(~24× per run on the v22 `paper_bandit_offpolicy --quick` smoke), so
this cost dominated wall time on otherwise-cheap configurations.

cProfile of one `paper_bandit_offpolicy --quick` run pre-v23 (cell=1,
algorithm=`offline_bandit_ipw`, behaviour=`reward_aligned`, alpha=4):

```
wall: 12.37s
ncalls  cumtime  function
     1   11.97s  _run_offline
    24    8.86s  _make_env_with_n          ← 72% of wall
    24    4.94s  _load_or_build_transition
    24    3.88s  _load_or_build_reward
```

## The fix

Both tensors depend only on the global `LATENT_STATES` / `OBS_STATES`
/ `N_ACTIONS` constants — not on `cell`, `alpha_conf`, or any
per-instance state.  v23 adds module-level lazy caches
(`_TRANSITION_CACHE`, `_REWARD_CACHE`) populated on first build and
shared across instances thereafter.

Sharing is safe.  Both perturbation paths
(`perturb_tabular_transition`, `perturb_tabular_rewards`) `.clone()`
the input tensor before modifying it; the only two mutation sites
in the codebase (`runner.py:878`, `tabular_sepsis.py:302`) are
reassignments of the form `self.transition = perturb_*(self.transition,
...)`, never in-place ops.  The defensive
`test_perturbation_does_not_mutate_cache` test catches any future
regression that breaks this invariant.

Custom `transition_path` / `reward_path` arguments still load from
disk and bypass the cache.

## Before/after timing

Same fixture (cell=1, `offline_bandit_ipw`, `reward_aligned`, α=4,
`total_frames=2000`, 10 train ckpts × 5 eval ckpts), fresh Python
process per run on CPU:

| | wall time |
|---|---|
| v22 (no cache) | **8.70 s** |
| v23 (cache)    | **2.14 s** |

**~4× speedup on bandit-shaped runs.**  Larger configurations
(`paper.yaml`, `coverage_stress.yaml`) see the same absolute saving
(~6 s / run) but a smaller relative one because their per-run cost
is dominated by the actual learner work rather than env construction.

## CSV invariance

The cached tensors are bit-identical to the freshly-built tensors —
the cache returns the same object the previous build path produced.
No semantic change to any output, including `eval.csv`, `train.csv`,
`summary.json`, or any figure generated by `make_all_figures.py`.

## Out of scope

* Reusing one eval env across checkpoints via `reset()` instead of
  reconstruction.  More invasive change touching the eval path; left
  for a future PR if this turns out to be insufficient.
* Caching the propensity model fit (also reconstructed per run, but
  far cheaper than the env tensors at ~1.5 s — diminishing returns).
* Caching for `ContinuousWardEnv` or other envs that don't have the
  same Python-loop overhead.
