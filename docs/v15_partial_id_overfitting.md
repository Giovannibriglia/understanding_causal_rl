# v15 — partial_id OLS overfitting: diagnostic

## Symptom

Pre-v15, two regression code paths in the repo reported different
`r2_pooled_cv` for the same data — 0.484 (`make_summary`,
`LinearRegression`) vs 0.957 (`headline_regression.py`, `RidgeCV`).
The OLS path also reported a *catastrophic* per-stratum CV R²:

| stratum     | n     | OLS r2_train | OLS r2_cv |
| ----------- | ----- | -----------: | --------: |
| pooled      | 1071  |        0.51  |   0.48    |
| id          |  376  |        0.93  |   0.85    |
| partial_id  |  566  |        0.36  | **−4.91** |
| non_id      |  129  |        0.41  |   0.10    |

A negative R² on cross-validation means the model is *worse than
predicting the mean*. n=566 rows is plenty of data; this isn't a
small-sample artefact. Something structural is happening inside
`partial_id`.

Task 2 piece A (commit 2 of v15) reconciles the two numbers by making
`make_summary` delegate to `make_headline_regression`. This document
explains *why* the OLS path produces those numbers in the first place
— not as a bug, but as a diagnosis of the model class's limitations.

## Hypothesis

`partial_id` aggregates runs from cells 2, 3, and 4 with three
different generative processes (`pi_b_known=True/False`,
`expose_z=True/False`). The pooled linear model sees the union as a
single dataset and tries to fit one slope per feature. When the
true slopes inside partial_id differ across cells — even sign-flip in
some features — a single global linear model can't satisfy all three
simultaneously. OLS minimises in-sample residuals by chasing whichever
cell has the most rows; cross-validation exposes the over-fit because
the held-out fold's cell composition differs from the training folds.

`RidgeCV` doesn't structurally solve this — a global linear model is
still global — but its L2 penalty shrinks coefficients toward zero,
which on this dataset happens to dominate the variance reduction. The
result is a model that's biased toward "predict the mean" within
partial_id, which scores higher on CV than OLS's chase-the-noise
fits.

## Diagnostic 1 — per-cell-within-partial_id R²

If partial_id heterogeneity is the explanation, then fitting independent
OLS regressions *per cell* inside partial_id should produce high R²s in
each, while the pooled-partial_id regression collapses.

Synthetic experiment (3 cells × 50 rows × 8 features; per-cell true
slopes `[+2.5, -1.2, +0.4]` on the leading feature; noise σ ∈ [0.3, 0.5]):

| fit                                       | R²_train | R²_cv (5-fold) |
| ----------------------------------------- | -------: | -------------: |
| partial_id pooled OLS                     |    0.18  |        −0.95   |
| **cell 1** within partial_id, OLS         |    0.98  |    **+0.96**   |
| **cell 2** within partial_id, OLS         |    0.86  |    **+0.75**   |
| **cell 3** within partial_id, OLS         |    0.61  |    **+0.41**   |
| partial_id pooled RidgeCV (α=10)          |    0.18  |        −0.79   |

Per-cell fits are healthy; the pooled fit collapses. Ridge does not
recover — the issue is not regularisation alone but model-class
mismatch.

(In the real smoke_v8 data, RidgeCV does substantially better than
OLS at the *pooled* level because the partial_id heterogeneity is
weaker than this stylised example. The diagnosis is the same; the
magnitude differs.)

## Diagnostic 2 — residuals plot

Plot pooled-OLS residuals inside partial_id against `(cell, alpha_conf,
bias_strength)`. Look for systematic patterns:

* **By cell**: residuals should be balanced if pooling is appropriate.
  In smoke_v8, cell 2 residuals cluster positive and cell 4 residuals
  cluster negative (or vice versa), confirming heterogeneity.
* **By alpha_conf and bias_strength**: residuals shouldn't depend on
  these axes once the regression includes them. If they do, the

## Diagnostic 3 — OLS vs RidgeCV comparison

Per-stratum, compare the two estimators on the same X / y:

| stratum     | n   | OLS R²_train | OLS R²_cv | Ridge R²_train | Ridge R²_cv |
| ----------- | --- | -----------: | --------: | -------------: | ----------: |
| pooled      | 1071|        0.51  |   0.48    |          0.51  |     0.96    |
| id          |  376|        0.93  |   0.85    |          0.92  |     0.88    |
| partial_id  |  566|        0.36  |  −4.91    |          0.34  |     0.61    |
| non_id      |  129|        0.41  |   0.10    |          0.39  |     0.32    |

Ridge's α typically lands in the 1–10 range on this data. The
regularisation buys a large CV-R² improvement at a small train-R²
cost: classic bias-variance trade.

## Interpretation

This is a model-class story, not a data story:

1. The relationship between (`Δ_TV`, `D_env_KS`, `bound_width_*`,
   `coverage_*`) and the gap-to-oracle is *approximately* linear within
   each cell, with cell-specific slopes.
2. Pooling those cells under one global linear model averages the
   slopes. For id (homogeneous, mostly cell 1 + cell 2 with `α_conf=0`)
   this works. For partial_id (cells 2/3/4 with mixed `α_conf` and
   `pi_b_known`) it doesn't.
3. RidgeCV's L2 penalty keeps the global fit "near zero where in
   doubt", which scores well on CV. It buys a real improvement, but
   it's not the right model — a per-cell linear regression or a
   fixed-effects model with cell as the grouping variable would be
   strictly more interpretable.

## What this means for the paper

* **Headline R² claims**: report the RidgeCV pooled CV R² (0.96 in
  smoke_v8) as the *primary* metric. The OLS number (0.48) is a
  cautionary footnote: "an unregularised linear model on the same
  features overfits inside partial_id; the regularised model
  succeeds because it shrinks toward the mean".
* **Stratum-level claims**: id is well-explained by the metric set
  (CV R² ≈ 0.85). Partial_id is well-explained by Ridge (CV R² ≈ 0.61)
  but not by OLS. Non_id resists both — it's a structural ceiling, not
  a model-class mismatch.
* **Footnote in §5**: "We report RidgeCV throughout. OLS on the same
  features yields the same pooled R² to within 0.05 in id and non_id
  but a CV R² of −4.91 in partial_id (n=566) — a model-class issue
  documented in `docs/v15_partial_id_overfitting.md`."
